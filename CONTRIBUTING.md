# دليل المساهمة

**نواة نظام حجز المواعيد – Booking Engine Core**

شكرًا لاهتمامك بالمساهمة في **Booking Engine Core** 🌟
هذا المشروع هو **نواة منطقية مفتوحة المصدر**، مصمَّمة لتكون مستقلة عن أي إطار عمل، وقابلة للتوسعة، وتركّز على منطق الأعمال فقط.

يرجى قراءة هذا الدليل بعناية قبل البدء بالمساهمة.

---

## 🎯 فلسفة المشروع

هذا المستودع يمثل **نواة (Core Engine)** وليس تطبيقًا جاهزًا.

### المبادئ الأساسية

-   التصميم القائم على النطاق (Domain-first)
-   عدم الاعتماد على أطر العمل
-   عدم تضمين منطق البنية التحتية
-   قواعد عمل واضحة وصريحة
-   التوسعة عبر الإضافات لا التعديل المباشر
-   البساطة قبل كثرة الميزات

أي مساهمة تخالف هذه المبادئ غالبًا لن يتم قبولها.

---

## ✅ المساهمات المقبولة

### ✔ مرحب بها

-   إصلاح الأخطاء
-   تحسين منطق النطاق الحالي
-   إعادة هيكلة لتحسين الوضوح بدون تغيير السلوك
-   زيادة تغطية الاختبارات
-   تحسين الوثائق
-   تحسين الأداء ضمن منطق النطاق
-   تحسينات صغيرة ومتوافقة مع خارطة الطريق

---

### ⚠ مقبولة بشروط

-   ميزات جديدة **مذكورة صراحة في ملف ROADMAP**
-   تحسينات لا تُدخل:

    -   ارتباطًا بأطر عمل
    -   منطق بنية تحتية
    -   سلوكيات SaaS

---

## ❌ ما هو خارج النطاق تمامًا

المساهمات التالية **لن تُقبل**:

-   واجهات REST أو GraphQL
-   أطر عمل ويب (FastAPI، Django، Flask…)
-   أنظمة مصادقة أو تفويض جاهزة
-   قواعد بيانات أو ORM
-   واجهات مستخدم
-   فوترة أو اشتراكات SaaS
-   مهام خلفية أو Queues

هذه الأمور يجب أن تُبنى **فوق النواة** وليس داخلها.

---

## 🧩 الحدود المعمارية

يجب احترام الطبقات التالية:

```
Domain (Entities, Value Objects)
Use Cases (منطق التطبيق)
Policies (الإعدادات والقواعد)
Extensions (Hooks)
Repositories (واجهات أو تخزين مؤقت فقط)
```

🚫 لا تجاوز لقواعد النطاق
🚫 لا خدمات ذات حالة (Stateful Services)
🚫 لا تسريب لمنطق البنية التحتية

---

## 🧪 متطلبات الاختبارات

كل مساهمة **يجب أن تحتوي على اختبارات**.

### قواعد الاختبار

-   الاختبارات يجب أن تكون ثابتة (Deterministic)
-   الأفضلية لاختبارات النطاق
-   تجنّب محاكاة (Mocking) منطق النطاق
-   ميزة جديدة → اختبارات جديدة
-   إصلاح خطأ → اختبار يمنع الرجوع

أي Pull Request بدون اختبارات لن يتم دمجه.

---

## 🧱 معايير كتابة الكود

-   Python ≥ 3.10
-   كود واضح وسهل القراءة
-   أسماء تعبّر عن لغة النطاق
-   تجنب التجريد المبكر
-   تجنب التأثيرات الجانبية الخفية
-   خدمات النطاق يجب أن تكون Stateless

إضافة Type Hints مستحبة وقد تصبح إلزامية مستقبلًا.

---

## 🔌 نظام الإضافات (Extensions)

عند العمل على الإضافات:

-   يجب عزل كل إضافة عن الأخرى
-   أخطاء الإضافات لا يجب أن تكسر التدفق الأساسي
-   لا يوجد اعتماد مباشر بين الإضافات
-   لا يجوز تعديل الكيانات الأساسية بشكل غير متوقع

---

## 🗺️ الالتزام بخارطة الطريق

قبل اقتراح ميزة جديدة:

1. راجع ملف `ROADMAP.md`
2. إن لم تكن الميزة موجودة → افتح Issue للنقاش أولًا
3. التغييرات الكبيرة تتطلب موافقة مسبقة

حتى لو كانت الميزة جيدة، قد تُرفض إذا لم تتماشى مع الرؤية.

---

## 📦 قواعد الـ Commit و Pull Request

### الرسائل

-   واضحة ومباشرة
-   كل Commit لتغيير منطقي واحد

مثال:

```
تحسين منطق كشف تعارض الحجوزات
```

---

### Pull Requests

-   وضّح سبب التغيير
-   اربط بـ Issues إن وجدت
-   اشرح تأثير التغيير
-   اذكر صراحة إن لم يتغير السلوك

---

## 🤝 سلوك المجتمع

-   الاحترام المتبادل
-   النقاش البنّاء
-   النقاش قبل التغييرات الكبيرة
-   الجودة أهم من السرعة

هذا المشروع يقدّر **الانضباط الهندسي** أكثر من كثرة الميزات.
