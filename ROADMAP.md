# 🗺️ خارطة الطريق – نواة نظام حجز المواعيد

**Booking Engine Core**

توضّح هذه الخارطة خطة تطوير **نواة نظام حجز مواعيد مرنة** كنظام مفتوح المصدر (Open Source Core)، مستقل عن أي إطار عمل، ومصمَّم ليكون أساسًا تُبنى عليه أنظمة ومنصات أخرى.

تركّز الخارطة على:

-   جودة النواة
-   سلامة التصميم
-   قابلية التوسعة
-   تجربة المطور

مع استبعاد متعمّد لأي عناصر تخص SaaS أو الواجهات.

---

## ✅ الإصدار 1.0 – النواة الأساسية (منتهي)

**الحالة:** ✅ مستقر
**الوسم:** `v1.0.0-core`

### نطاق الإصدار

تقديم نواة حجز مكتملة من ناحية المنطق الأساسي وقابلة للتوسعة.

### المزايا

-   نموذج نطاق (Domain Model) واضح للحجز مع دورة حياة صريحة
-   حالات الاستخدام الأساسية:

    -   إنشاء حجز
    -   إلغاء حجز
    -   إعادة جدولة حجز

-   قواعد الحجز:

    -   التحقق من التوفر
    -   كشف التعارضات

-   سياسات حجز قابلة للإعداد (Policies)
-   نظام إضافات (Hooks) قبل/بعد كل حالة استخدام، معزول حسب السياق (Context)
-   صلاحيات أساسية اختيارية
-   مستودعات بيانات داخل الذاكرة (In-memory Repositories)
-   اختبارات كاملة على مستوى النطاق (Domain-level Tests)

### خارج النطاق (بشكل متعمّد)

-   طبقة API
-   قواعد بيانات أو تخزين دائم
-   أنظمة مصادقة
-   واجهات مستخدم
-   منطق SaaS أو فوترة

---

## 🟢 الإصدار 1.1 – التثبيت وتجربة المطور

**الهدف:** تحسين الوضوح والأمان دون تغيير السلوك الأساسي للنواة.

### التحسينات المخطط لها

-   إضافة Type Hints أقوى لجميع الواجهات العامة
-   توضيح حدود الـ Public API
-   تحسين وتوحيد أسماء الوحدات والدوال
-   تحسين الوثائق والأمثلة
-   تحسينات داخلية بسيطة بدون تغييرات جذرية

---

## 🟡 الإصدار 2.0 – نضج نموذج النطاق

**الهدف:** تعزيز التعبير الدوميني (Domain Expressiveness) ودقة الأخطاء.

### التحسينات المخطط لها

-   تحويل Context إلى Value Object قوي:

    -   `__repr__`
    -   `__eq__`

-   إضافة استثناءات خاصة بالنطاق:

    -   ContextNotFoundError
    -   ContextAlreadyExistsError
    -   BookingRuleViolationError

-   منع تسجيل Context مكرر
-   تحويل حالة الحجز من `string` إلى `enum` (مع الحفاظ على التوافق)

---

## 🟡 الإصدار 2.1 – الإعدادات والأمان

**الهدف:** تحسين قابلية الإعداد بدون تلويث النواة.

### التحسينات المخطط لها

-   كائن إعدادات صريح لكل Context
-   طبقة PolicyResolver
-   دعم التخزين المؤقت للسياسات لكل Context
-   توسيع واجهات المستودعات:

    -   `list_by_context_and_service`

---

## 🟡 الإصدار 2.2 – تحسين نظام الإضافات (Hooks)

**الهدف:** جعل الإضافات أكثر أمانًا ومرونة.

### التحسينات المخطط لها

-   عزل الأخطاء بين الإضافات
-   دعم أولويات التنفيذ (اختياري)
-   تحسين توثيق نظام الإضافات
-   نموذج تنفيذ أكثر أمانًا للإضافات

---

## 🟡 الإصدار 2.3 – الصلاحيات والتحكم بالوصول

**الهدف:** توسيع نظام الصلاحيات بدون ربطه بأنظمة مصادقة محددة.

### التحسينات المخطط لها

-   RoleRepository
-   مجموعات صلاحيات (Permission Groups)
-   سياسات صلاحيات لكل حالة استخدام
-   دعم توسيع الصلاحيات عبر Hooks

---

## 🟡 الإصدار 2.4 – الاختبارات والموثوقية

**الهدف:** زيادة الثقة في النظام تحت الحالات الحافة (Edge Cases).

### التحسينات المخطط لها

-   Property-based Testing
-   اختبارات الأداء
-   سيناريوهات ضغط وتعقيد عالية للتعارضات

---

## 🔵 أفكار مستقبلية (غير مجدولة)

هذه الأفكار معروفة ولكن لم يتم جدولتها للحفاظ على التركيز:

-   Application Layer و Dependency Injection
-   تنفيذ غير متزامن للإضافات
-   Observability (Logs / Metrics / Tracing)
-   SDKs (Python / JavaScript)
-   أي منطق SaaS أو إدارة اشتراكات

---

## 🚫 ما لن يكون جزءًا من النواة أبدًا

-   واجهات مستخدم
-   أطر عمل API
-   أنظمة مصادقة جاهزة
-   فوترة أو اشتراكات
-   لوحات تحكم SaaS

هذه العناصر يجب أن تُبنى **فوق النواة** وليس داخلها.
